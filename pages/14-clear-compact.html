/clear and /compact

Claude's context window is finite. /clear resets it completely. /compact summarizes and compresses it — keeping important decisions but clearing the bulk. Knowing when to use each prevents context overflow.

🗄️
The Desk Analogy

After hours of work, your desk is covered in notes, printouts, sticky notes, and half-finished documents. You have two options:

  • /compact — take a photo of the important stuff, file the rest. Your desk gets clean but you still know what mattered.
  • /clear — sweep everything off. Completely fresh. Whatever was there is gone.

Context window management is exactly that tradeoff: preserve continuity vs start fresh.

Use when:
Context is getting full mid-task, you want to continue from where you are, long-running sessions (Ralph loop)

Hands-On Exercise

01

Build up context

Start a Claude Code session and ask it to write 5+ files or analyze a codebase. Build up significant context over 10+ messages.

02

Check context usage

Ask: "How full is your context window?" — note the percentage. Most sessions will show it filling up after significant work.

03

Run /compact

Type /compact. Watch Claude summarize. Then ask "What do you remember from our session?" — verify the key decisions were kept.

04

Try /clear on a new task

Start a different task. Type /clear first to reset context. Notice the difference — Claude has no memory of the previous session.

⚡ Key Takeaways

✓
Context window = limited RAM. It fills up during long sessions.
✓
/compact = smart compression. Keeps key decisions, drops chatter. Drops to ~20-30%.
✓
/clear = full reset. Drops to 0%. Use when switching tasks entirely.
✓
Autonomous agents (like Ralph) use /compact automatically at ~80% to keep running.
