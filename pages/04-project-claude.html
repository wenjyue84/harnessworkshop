Project CLAUDE.md

Every project has a different stack, a different team, different rules. Project-level CLAUDE.md gives Claude the context it needs to work in THIS specific codebase.

The Workplace Employee Handbook

Think of it as the employee handbook for a specific Prisma client project. The global CLAUDE.md is "how this developer generally behaves at Prisma." The project CLAUDE.md is "how this specific system works." The project overrides when there is conflict.

Without a project CLAUDE.md, Claude works with general knowledge of your stack. With one, Claude knows the exact MQTT topic structure for Rui Sin Plastic's injection machines, which WhatsApp message types Pelangi Hostel's chatbot handles, or which EPF/SOCSO contribution table applies to the prisma-payroll system. This is the difference between a good contractor and someone who knows your codebase inside out.

💡
More specific = more useful. A sparse CLAUDE.md is better than none. A detailed one — with anti-patterns and architecture decisions — makes Claude dramatically more accurate.

What a Strong Project CLAUDE.md Looks Like

./CLAUDE.md (Prisma IoT Monitor — Rui Sin Plastic)
# iot-monitor/CLAUDE.md — Rui Sin Plastic Injection Machine Monitor

## Tech Stack
- Node.js + Express REST API
- MQTT (Mosquitto broker) for sensor data ingestion
- Supabase (PostgreSQL) for time-series storage
- n8n for alert automation workflows

## IoT Data Conventions
- Sensor payload format: { deviceId, timestamp, value, unit, shiftId }
- MQTT topics follow: prisma/ruisin/{machineId}/{sensorType}
- Timestamps always in ISO 8601 UTC; display in MYT (UTC+8)

## What To Avoid
- Do NOT hardcode MQTT broker credentials — use process.env.MQTT_PASSWORD
- Do NOT use blocking I/O in MQTT message handlers — always async
- Do NOT modify shift schedule logic without checking with Rui Sin ops team

## Payroll Module (prisma-payroll)
- EPF: Employee 11%, Employer 13% (standard). Refer config/epf-rates.json
- SOCSO + EIS contributions from config/socso-table.json (LHDN 2025 rates)
- CP21/CP22 forms generated via src/lhdn/cp-forms.js — never modify schema

## AI Chatbot Module (ai-chatbot)
- WhatsApp via Baileys library; Telegram via node-telegram-bot-api
- Support Bahasa Melayu, English, and Mandarin responses
- Session state stored in Supabase, not in-memory — stateless restarts

Hands-On Exercise

Exercise 04 — Write a Project CLAUDE.md10 min
1
Open a Prisma project folder
Use one of: iot-monitor, ai-chatbot, or prisma-payroll. If starting from scratch: mkdir iot-monitor && cd iot-monitor && git init
2
Create CLAUDE.md in the root
touch CLAUDE.md or open a new file in your editor at the project root.
3
Add these four sections
Tech Stack, IoT/Chatbot/Payroll Conventions, File Structure, and What To Avoid. Include the sensor payload format or LHDN rates reference as appropriate.
4
Add an anti-patterns section
Write at least 2-3 things Claude should never do. For IoT: "Never hardcode MQTT credentials." For payroll: "Never modify CP21 form schema." Anti-patterns are the most valuable lines in any CLAUDE.md.
5
Run Claude and verify context
Run claude in the project and type /status. Ask Claude "what stack are we using?" and "what is the MQTT topic format?" — it should answer correctly from your CLAUDE.md.

Key Takeaways

📁
CLAUDE.md in repo root = project instructions. It must be committed to git — this is version-controlled documentation, not a secret.
📈
Overrides global rules for this project. When there is a conflict between global and project rules, project wins. Use this deliberately.
⚠
Anti-patterns are the most valuable content. Telling Claude what NOT to do prevents the most costly mistakes. Always include a "What To Avoid" section.
