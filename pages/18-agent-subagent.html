Agent / Subagent

Agents are not roles or personas. They are systems that engineer exactly what tokens sit in the context window at each step. Split large tasks: main agent coordinates, subagents execute in isolated contexts.

"Context engineering is the delicate art and science of filling the context window with just the right information for the next step." — Andrej Karpathy, June 2025
🏗️
The Project Manager Model

A project manager doesn't personally write every line of code, conduct every interview, and draft every email. They spawn specialists: the coder handles the implementation, the researcher handles the documentation, the writer handles the copy. Each specialist has their own focused workspace.

The PM synthesizes results. That's the agent/subagent pattern. One context window coordinates; multiple isolated context windows execute. The main agent stays lean. Subagents do the heavy lifting without polluting each other's context.

🧠 Context Window Architecture
Main Agent (Orchestrator)
Holds: plan, high-level state, task list, synthesized results
Context: 25% used (stays lean)
↓ spawns
Subagent A
Task: Auth module analysis
Context: 82% — heavy lifting
Subagent B
Task: DB schema migration
Context: 76% — heavy lifting
Subagent C
Task: Write test suites
Context: 68% — heavy lifting
↑ returns results
Main agent synthesizes findings without ever loading the raw work into its context
💡
The key insight: Subagent isolation means each agent gets only what it needs. No cross-contamination. Main agent stays lean for coordination. This is how you scale beyond a single context window.

How Claude Code spawns subagents

Claude Code has a built-in Task tool that spawns subagents. You don't need to configure anything — it's available by default.

// Claude Code's internal Task tool — orchestrator uses this: Task({ description: "Analyze auth module for vulnerabilities", prompt: "Read /src/auth/* thoroughly. Find all security vulnerabilities. Return structured JSON findings with severity, file, line number.", subagent_type: "general-purpose" }) // The subagent: // - Gets its own isolated context window // - Runs the task independently // - Returns structured findings // - Main agent never loads the raw /src/auth/* files

Hands-On Exercise

01

Choose a task too big for one context

Pick something large: "Analyze my entire codebase for dead code", "Review all API endpoints for missing error handling", "Audit all SQL queries for injection risks".

02

Break it into parallel subtasks

Ask Claude: "Break this task into 3-5 independent subtasks that could each run in their own context window."

03

Ask Claude to orchestrate

Say: "You are the orchestrator. Spawn subagents for each subtask using the Task tool, then synthesize their findings."

04

Review the synthesized result

Compare the quality vs asking Claude to do the entire task in a single context. Note how much more thorough the subagent approach is — each specialist went deep on their slice.
