Agent Team

A single agent makes mistakes without accountability. A team of specialized agents — Coder, Tester, Reviewer — creates adversarial pressure. The Tester breaks what the Coder builds; the Reviewer critiques what both produce.

👥
The Code Review Effect

One developer reviewing their own code misses ~60% of bugs — they read what they meant to write, not what they actually wrote. Add a second developer for code review and the catch rate triples. Add a dedicated QA engineer and it triples again.

Agent teams replicate this dynamic. The Coder is optimistic — focused on shipping. The Tester is adversarial — their job is to break things. The Reviewer is balanced — critiques both. When these three collaborate, quality is enforced structurally, not by hoping one agent is careful enough.

System prompts for each role
// Coder: optimistic, focused on shipping "You are the Coder. Implement clean, working code for the given task. Write idiomatic TypeScript with proper types. When done, describe what you implemented and hand off to the Tester." // Tester: adversarial, focused on breaking "You are the Tester. Your JOB is to break the Coder's implementation. Write tests for: happy paths, edge cases, null/undefined, empty arrays, concurrent calls, error conditions. Report ALL failures with specific reproduction cases." // Reviewer: balanced, focused on quality "You are the Reviewer. Read the code AND the test results critically. Flag: poor naming, missing error handling, security issues, performance problems, incomplete test coverage. Approve ONLY when both code quality and test coverage meet senior engineer standards."

Hands-On Exercise

01

Pick a function to implement

Choose something non-trivial: parseCSV(string) → Record[], validatePassword(string) → {valid, reasons}, or deepMerge(obj1, obj2) → merged.

02

Simulate the Coder role

Ask Claude: "Act as the Coder. Implement [function] in TypeScript with proper error handling."

03

Switch to Tester

Ask Claude: "Now act as the Tester. Your job is to BREAK this implementation. Write tests for every edge case you can think of. What fails?"

04

Switch to Reviewer

Ask: "Act as the Reviewer. Critically evaluate the code AND the test results. What issues do you see? What's missing? Approve or reject with reasons."

05

Coder addresses feedback

Ask: "Act as the Coder again. Address all the Tester failures and Reviewer feedback." Compare the final version to the initial implementation — notice the quality jump.
