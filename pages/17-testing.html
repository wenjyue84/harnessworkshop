Test Types That Guide Claude Best

U
Unit tests — exact input/output expectations. The clearest signal for Claude.
I
Integration tests — end-to-end flows. Catches hidden coupling bugs.
E
Edge cases — nulls, empty arrays, negative numbers, huge inputs. Makes the spec airtight.
T
Type tests — TypeScript compilation passes. Prevents an entire class of runtime errors.

Exercise: Write Tests First

TDD with Claude

1

Pick a Prisma payroll function

Use calculateEPF(grossSalary, age): employee pays 11% (max RM5k), employer 13% if salary ≤ RM5k else 12%. Age 60+ = different rate.

2

Write 5 test cases first

Test: salary RM3,000 / exactly RM5,000 / RM8,000 / negative salary throws / employee age 60+. Write these BEFORE asking Claude to implement.

3

Ask Claude to implement

"Implement calculateEPF(grossSalary, age) so all 5 tests pass. Use KWSP 2025 contribution tables."

4

Watch Claude iterate

Claude runs tests, reads LHDN table failures, updates the contribution logic, repeats. Payroll bugs cost real money — the test harness is Prisma's safety net.

5

Add SOCSO + EIS edge cases

Add 3 edge cases for calculateSOCSO() and calculateEIS(). Ask Claude to fix all failures. Every statutory deduction must be LHDN-compliant.

6

Reflect

When LHDN publishes new rates each year, you update the tests — Claude refactors the implementation to match. This is how Prisma Payroll stays compliant without manual audits.
